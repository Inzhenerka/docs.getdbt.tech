<DetailsToggle alt_header="Является ли dbt Semantic Layer открытым исходным кодом?">

<Constant name="semantic_layer" /> является проприетарным продуктом; однако некоторые компоненты <Constant name="semantic_layer" /> имеют открытый исходный код, например <Constant name="core" /> и MetricFlow.

Пользователи <Constant name="cloud" /> Developer или <Constant name="core" /> могут определять метрики в своём проекте, включая локальный проект <Constant name="core" />, используя <Constant name="cloud_ide" />, <Constant name="cloud_cli" /> или CLI <Constant name="core" />. Однако для работы с универсальным <Constant name="semantic_layer" /> и доступа к этим метрикам через API или сторонние инструменты пользователям необходимо иметь тариф <Constant name="cloud" /> [Enterprise или Enterprise+](https://www.getdbt.com/pricing/).

Обратитесь к разделу <a href="https://docs.getdbt.com/docs/cloud/billing">Billing</a> для получения дополнительной информации.

</DetailsToggle>

<DetailsToggle alt_header="Как пользователи с открытым исходным кодом могут использовать dbt Semantic Layer?">

<Constant name="semantic_layer" /> требует использования сервиса, предоставляемого <Constant name="cloud" />, для координации запросов. Пользователи open source, которые не используют <Constant name="cloud" />, в настоящее время могут обойти отсутствие сервисного слоя. Для этого они могут запустить команду `mf query --explain` в командной строке. Эта команда генерирует SQL-код, который затем можно использовать в существующих системах для выполнения и управления запросами.
  
По мере доработки API-слоёв MetricFlow некоторым пользователям может оказаться проще настроить собственные кастомные сервисные слои для управления запросами. В настоящий момент это не рекомендуется, так как границы API вокруг MetricFlow ещё недостаточно чётко определены для широкого использования сообществом.

</DetailsToggle>

<DetailsToggle alt_header="Почему мой запрос в Cloud CLI ограничен 100 строками?">

Значение `limit` по умолчанию для запросов, выполняемых из <Constant name="cloud_cli" />, составляет 100 строк. Мы установили это значение, чтобы предотвратить возврат излишне больших наборов данных, поскольку <Constant name="cloud_cli" /> обычно используется для запросов к dbt Semantic Layer в процессе разработки, а не для продакшн-отчётности или доступа к большим объёмам данных. В большинстве рабочих процессов достаточно получить лишь подмножество данных.
  
При необходимости вы можете изменить это ограничение, указав параметр `--limit` в запросе. Например, чтобы вернуть 1000 строк, можно выполнить команду `dbt sl list metrics --limit 1000`.

</DetailsToggle>

<DetailsToggle alt_header="Могу ли я ссылаться на запросы MetricFlow внутри моделей dbt?">

dbt полагается на макросы Jinja для компиляции SQL, в то время как MetricFlow основан на Python и выполняет прямую рендеринг SQL, ориентированную на конкретный диалект. MetricFlow не поддерживает сквозную рендеринг макросов Jinja, поэтому мы не можем легко ссылаться на запросы MetricFlow внутри моделей dbt.

Помимо технических сложностей, которые можно было бы преодолеть, мы видим метрики как конечный узел вашего DAG и место для потребления метрик пользователями. Если вам нужно выполнить дополнительную трансформацию поверх метрики, это обычно является признаком того, что требуется больше моделирования.

</DetailsToggle>

<DetailsToggle alt_header="Могу ли я создавать таблицы в моей платформе данных с помощью MetricFlow?">

Вы можете использовать предстоящую функцию **Exports**, которая позволит вам создать **предопределённый** запрос MetricFlow (см. [saved queries](/docs/build/saved-queries)) в виде таблицы на вашей платформе данных. Эта функция будет доступна только для клиентов <Constant name="cloud" />. Это связано с тем, что MetricFlow в первую очередь предназначен для генерации SQL‑запросов, тогда как отправка соответствующего запроса и выполнение любых DDL‑операций относится к зоне ответственности сервисного слоя, работающего поверх MetricFlow.

</DetailsToggle>

<DetailsToggle alt_header="Как мне перейти с устаревшего Semantic Layer на новый?">

Если вы используете устаревший Semantic Layer, мы настоятельно рекомендуем вам [обновить вашу версию dbt](/docs/dbt-versions/upgrade-dbt-version-in-cloud) до dbt v1.6 или выше, чтобы использовать новый dbt Semantic Layer. Обратитесь к специальному [руководству по миграции](/guides/sl-migration) для получения дополнительной информации.

</DetailsToggle>

<DetailsToggle alt_header="Как вы храните мои данные?">

Данные пользователей проходят через Semantic Layer на пути обратно из хранилища. dbt Labs обеспечивает безопасность, аутентифицируясь через хранилище данных клиента. В настоящее время мы не кэшируем данные на длительный срок, но они могут временно оставаться в системе до 10 минут, обычно меньше. В будущем мы введем функцию кэширования, которая позволит нам кэшировать данные на нашей инфраструктуре до 24 часов.

</DetailsToggle>
