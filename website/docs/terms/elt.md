Извлечение, загрузка, преобразование (ELT) — это процесс, который начинается с извлечения данных из различных источников, затем загрузки их в целевое <Term id="data-warehouse" />, и, наконец, преобразования.

ELT стал парадигмой управления потоками информации в современном хранилище данных. Это представляет собой фундаментальный сдвиг от того, как ранее обрабатывались данные, когда <Term id="etl">Извлечение, преобразование, загрузка (ETL)</Term> была наиболее распространенной схемой обработки данных, используемой большинством компаний.

Переход от ETL к ELT означает, что вам больше не нужно захватывать ваши преобразования во время первоначальной загрузки данных в ваше хранилище данных. Вместо этого вы можете загрузить все ваши данные, а затем строить преобразования на их основе. Команды данных сообщают, что рабочий процесс ELT имеет несколько преимуществ по сравнению с традиционным рабочим процессом ETL, которые мы рассмотрим [подробно позже в этом глоссарии](#benefits-of-elt).

## Как работает ELT

В процессе ELT данные извлекаются из источников данных, загружаются в целевую платформу данных и, наконец, преобразуются для аналитического использования. Мы подробно рассмотрим три компонента (извлечение, загрузка, преобразование) здесь.

![Диаграмма, изображающая рабочий процесс ELT. Данные изображены как извлекаемые из примеров источников данных, таких как Email CRM, платформа Facebook Ads, бэкенд базы данных и Netsuite. Затем данные загружаются как необработанные данные в хранилище данных. Оттуда данные преобразуются внутри хранилища путем переименования, приведения типов, объединения или обогащения необработанных данных. Результатом являются смоделированные данные внутри вашего хранилища данных.](/img/docs/terms/elt/elt-diagram.png)

### Извлечение

В процессе извлечения данные извлекаются из нескольких источников данных. Извлекаемые данные, в основном, это данные, которые команды в конечном итоге хотят использовать для аналитической работы. Некоторые примеры источников данных могут включать:

- Базы данных бэкенд-приложений 
- Маркетинговые платформы 
- CRM для электронной почты и продаж 
- и многое другое!

Доступ к этим источникам данных с помощью вызовов API может быть сложной задачей для людей и команд, у которых нет технической экспертизы или ресурсов для создания собственных скриптов и автоматизированных процессов. Однако недавнее развитие определенных продуктов с открытым исходным кодом и программного обеспечения как услуги (SaaS) устранило необходимость в этой индивидуальной разработке. Установив возможность создания и управления конвейерами в автоматизированном режиме, вы можете извлекать данные из источников данных и загружать их в хранилища данных через пользовательский интерфейс.

Поскольку не каждый источник данных будет интегрироваться с инструментами SaaS для извлечения и загрузки, иногда неизбежно, что команды будут писать пользовательские скрипты для загрузки данных в дополнение к своим инструментам SaaS.

### Загрузка

На этапе загрузки извлеченные данные загружаются в целевое хранилище данных. Некоторые примеры современных хранилищ данных включают Snowflake, Amazon Redshift и Google BigQuery. Примеры других платформ хранения данных включают озера данных, такие как Databricks’s Data Lakes. Большинство SaaS-приложений, которые извлекают данные из ваших источников данных, также загружают их в ваше целевое хранилище данных. Индивидуальные или внутренние процессы извлечения и загрузки обычно требуют сильных навыков в области инженерии данных и технических навыков.

На этом этапе процесса ELT данные в основном остаются неизменными с момента их извлечения. Если вы используете инструмент извлечения и загрузки, такой как Fivetran, возможно, с вашими данными была проведена легкая нормализация. Но, по сути, данные, загруженные в ваше хранилище данных на этом этапе, находятся в своем необработанном формате.

### Преобразование

На последнем этапе преобразования необработанные данные, загруженные в ваше хранилище данных, наконец, готовы к моделированию! Когда вы впервые смотрите на эти данные, вы можете заметить несколько вещей о них…

- Имена столбцов могут быть неясными
- Некоторые столбцы могут иметь неправильный тип данных
- Таблицы не связаны с другими таблицами
- Временные метки могут быть в неправильном часовом поясе для вашей отчетности
- <Term id="json" /> поля могут потребовать развертывания
- Таблицы могут не иметь <Term id="primary-key">первичных ключей</Term>
- И многое другое!

...отсюда и необходимость в преобразовании! В процессе преобразования данные из ваших источников данных обычно:

- **Легко преобразуются**: Поля приводятся к правильным типам, временные метки унифицируются по часовым поясам, таблицы и поля переименовываются соответствующим образом и многое другое. 
- **Сильно преобразуются**: Добавляется бизнес-логика, устанавливаются соответствующие материализации, данные объединяются и т.д. 
- **Проверяются**: Данные тестируются в соответствии с бизнес-стандартами. На этом этапе команды данных могут убедиться, что первичные ключи уникальны, связи моделей соответствуют, значения столбцов корректны и многое другое.

Распространенные способы преобразования ваших данных включают использование современных технологий, таких как dbt, написание пользовательских SQL-скриптов, которые автоматизируются с помощью планировщика, использование хранимых процедур и многое другое.

## ELT против ETL

Основное различие между традиционным ETL и современным рабочим процессом ELT заключается в том, когда [преобразование данных](https://www.getdbt.com/analytics-engineering/transformation/) и загрузка происходят. В рабочих процессах ETL данные, извлеченные из источников данных, преобразуются до загрузки в целевые платформы данных. В новых рабочих процессах ELT данные преобразуются после загрузки в выбранную платформу данных. Почему это так важно?

|  | ELT | ETL |
|---|---|---|
| Необходимые навыки программирования | Часто требуется мало или совсем не требуется кода для извлечения и загрузки данных в ваше хранилище данных. | Часто требует пользовательских скриптов или значительных усилий в области инженерии данных для извлечения и преобразования данных перед загрузкой. |
| Разделение задач | Этапы извлечения, загрузки и преобразования могут быть явно разделены различными продуктами. | Процессы ETL часто заключены в одном продукте. |
| Распределение преобразований | Поскольку преобразования происходят последними, существует большая гибкость в процессе моделирования. Сначала беспокойтесь о том, чтобы собрать ваши данные в одном месте, затем у вас будет время исследовать данные, чтобы понять, как лучше их преобразовать. | Поскольку преобразование происходит до загрузки данных в целевое место, командам необходимо провести тщательную работу заранее, чтобы убедиться, что данные правильно преобразованы. Сильные преобразования часто происходят на более поздних этапах в слое BI. |
| [Распределение команд данных](https://www.getdbt.com/data-teams/analytics-job-descriptions/) | Рабочие процессы ELT позволяют членам команды данных, знающим SQL, создавать собственные конвейеры извлечения и загрузки и преобразования. | Рабочие процессы ETL часто требуют от команд более высокой технической квалификации для создания и поддержки конвейеров. |

Почему принятие ELT так быстро возросло в последние годы? Несколько причин:

- **Изобилие дешевого облачного хранения с современными хранилищами данных.** Создание современных хранилищ данных, таких как Redshift и Snowflake, позволило командам всех размеров хранить и масштабировать свои данные по более эффективной цене. Это стало огромным стимулом для рабочего процесса ELT.
- **Разработка инструментов для извлечения и загрузки данных с низким или нулевым кодом.** Продукты, которые требуют минимальной технической экспертизы, такие как Fivetran и Stitch, которые могут извлекать данные из множества источников и загружать их в различные хранилища данных, помогли снизить барьер для входа в рабочий процесс ELT. Команды данных теперь могут снизить часть нагрузки по инженерии данных, необходимую для извлечения данных и создания сложных преобразований.
- **Истинный кодовый слой преобразования с разработкой dbt.** До разработки dbt не существовало единого продукта для слоя преобразования. dbt помогает аналитикам данных применять лучшие практики программной инженерии (контроль версий, CI/CD и тестирование) к преобразованию данных, в конечном итоге позволяя любому, кто знает SQL, участвовать в процессе ELT.
- **Увеличение совместимости между слоями ELT и технологиями в последние годы.** С расширением слоев извлечения, загрузки и преобразования, которые тесно интегрируются друг с другом и с облачным хранилищем, рабочий процесс ELT стал более доступным, чем когда-либо. Например, Fivetran создает и поддерживает [пакеты dbt](https://hub.getdbt.com/), чтобы помочь писать преобразования dbt для источников данных, с которыми они подключаются.

## Преимущества ELT

Вы часто слышите о преимуществах рабочего процесса ELT для данных, но иногда забываете говорить о преимуществах, которые он приносит людям. Существует множество преимуществ, которые этот рабочий процесс приносит самим данным (которые мы подробно опишем ниже), такие как возможность воссоздания исторических преобразований, тестирования данных и моделей данных и многое другое. Мы также хотим использовать этот раздел, чтобы подчеркнуть возможности, которые рабочий процесс ELT предоставляет как членам команды данных, так и бизнес-стейкхолдерам.

### Преимущество ELT #1: Данные как код

Мы уже говорили об этом ранее: рабочий процесс ELT позволяет командам данных функционировать как программисты. Но что это действительно означает? Как это влияет на ваши данные?

#### Код аналитики теперь может следовать тем же лучшим практикам, что и программный код 

В своей основе преобразования данных, которые происходят последними в конвейере данных, позволяют выполнять преобразования на основе кода и с контролем версий. Эти два фактора позволяют членам команды данных:

- Легко воссоздавать исторические преобразования, откатывая коммиты 
- Устанавливать тесты на основе кода
- Реализовывать CI/CD рабочие процессы
- Документировать модели данных, как это делается с обычным программным кодом.

#### Масштабирование, сделанное устойчивым

По мере роста вашего бизнеса количество источников данных соответственно увеличивается. Таким образом, увеличивается и количество преобразований и моделей, необходимых для вашего бизнеса. Управление большим количеством преобразований без контроля версий или автоматизации не является масштабируемым.

Рабочий процесс ELT использует преобразования, происходящие последними, чтобы обеспечить гибкость и лучшие практики программной инженерии для преобразования данных. Вместо того чтобы беспокоиться о том, как ваши скрипты извлечения масштабируются по мере увеличения объема данных, данные могут извлекаться и загружаться автоматически всего за несколько кликов.

### Преимущество ELT #2: Дайте силу людям

Рабочий процесс ELT открывает мир возможностей для людей, работающих с этими данными, а не только для самих данных.

#### У empowers членов команды данных

Аналитики данных, инженеры аналитики и даже ученые данных больше не должны зависеть от инженеров данных для создания пользовательских конвейеров и моделей. Вместо этого они могут использовать продукты с графическим интерфейсом, такие как Fivetran и Airbyte, чтобы извлекать и загружать данные за них. 

Наличие преобразования в качестве последнего шага в рабочем процессе ELT также позволяет специалистам по данным использовать свои знания о данных и SQL, чтобы сосредоточиться больше на фактическом моделировании данных. 

#### Способствует большей прозрачности для конечных пользователей бизнеса

Команды данных могут предоставить код с контролем версий, используемый для преобразования данных для аналитики, конечным пользователям бизнеса, поскольку преобразования больше не скрыты в процессе ETL. Вместо того чтобы вручную отвечать на распространенный вопрос: «Как эти данные генерируются?», специалисты по данным могут направить пользователей бизнеса к документации и репозиториям. Участие конечных пользователей бизнеса или их просмотр преобразований данных способствует большему сотрудничеству и осведомленности между бизнесом и специалистами по данным.

## Инструменты ELT

Как уже упоминалось, недавнее развитие определенных технологий и продуктов помогло снизить барьер для входа в реализацию рабочего процесса ELT. Большинство из этих новых продуктов действуют как одна или две части процесса ELT, но некоторые охватывают все три части. Мы опишем некоторые из текущих инструментов в экосистеме ELT ниже.

| Продукт | E/L/T? | Описание | Открытая версия? |
|---|---|---|---|
| Fivetran/HVR | E, немного T, L | Fivetran — это компания SaaS, которая помогает командам данных извлекать, загружать и выполнять некоторые преобразования с их данными. Fivetran легко интегрируется с современными хранилищами данных и dbt. Они также предлагают преобразования, которые используют dbt Core. | :x: |
| Stitch от Talend | E, L | Stitch (часть Talend) — это еще один продукт SaaS, который имеет множество соединителей данных для извлечения данных и загрузки их в хранилища данных. | :x: |
| Airbyte | E, L | Airbyte — это облачный сервис с открытым исходным кодом, который позволяет командам создавать конвейеры извлечения и загрузки данных. | :white_check_mark: |
| Funnel | E, немного T, L | Funnel — это еще один продукт, который может извлекать и загружать данные. Соединители данных Funnel в основном сосредоточены на источниках данных в области маркетинга. | :x: |
| dbt | T | dbt — это инструмент преобразования, который позволяет аналитикам и инженерам данных преобразовывать, тестировать и документировать данные в облачном хранилище данных. dbt предлагает как открытый, так и облачный продукт. | :white_check_mark: |

## Заключение

Последние несколько лет были бурными для мира данных. Увеличение доступности и доступности облачных хранилищ, инструменты для извлечения и загрузки данных без кода и истинный слой преобразования с помощью dbt позволили рабочему процессу ELT стать предпочтительным рабочим процессом для аналитики. ETL предшествует ELT и отличается тем, когда данные преобразуются. В обоих процессах данные сначала извлекаются из различных источников. Однако в процессах ELT данные загружаются в целевую платформу данных, а затем преобразуются. Рабочий процесс ELT в конечном итоге позволяет членам команды данных извлекать, загружать и моделировать свои данные гибким, доступным и масштабируемым образом.
