<Constant name="cloud" /> предлагает разнообразные [тарифы и варианты ценообразования](https://www.getdbt.com/pricing/), которые подходят под потребности вашей организации. Благодаря гибким вариантам оплаты, удобным как для крупных предприятий, так и для малого бизнеса, а также [доступности серверов](/docs/cloud/about-cloud/access-regions-ip-addresses) по всему миру, <Constant name="dbt_platform" /> — это самый быстрый и простой способ начать трансформацию ваших данных.

## Как работает ценообразование dbt?

Как клиент, вы платите за количество мест и объем использования, потребленный каждый месяц. Места оплачиваются в основном на основе количества приобретенных лицензий Developer и Read.

Использование основывается на количестве [Successfully Models Built](#what-counts-as-a-successful-model-built) и, если приобретено и используется, <Constant name="semantic_layer" /> [Queried Metrics](#what-counts-as-a-queried-metric), с учётом разумного использования. Все расчёты биллинга выполняются в координированном универсальном времени (UTC).

### Что считается лицензией рабочего места (seat)?
Подробнее о распределении пользователей в вашем аккаунте можно узнать в разделе [Users and licenses](/docs/cloud/manage-access/seats-and-users).
Существует четыре возможных типа лицензий рабочего места:
* **Analyst** &mdash; для наборов прав доступа, которые назначаются и могут совместно использоваться пользователями, которым не требуется ежедневный доступ. Требует покупки лицензии разработчика.
* **Developer** &mdash; для наборов прав доступа, которым необходимо ежедневное взаимодействие с <Constant name="dbt_platform" />.
* **IT** &mdash; для доступа к определённым функциям, связанным с управлением аккаунтом (например, настройка интеграции с git).
* **Read-Only** &mdash; для доступа к просмотру определённых документов и отчётов.

### Что считается успешно построенной моделью?

<Constant name="cloud" /> считает **успешно построенной моделью (Successful Model Built)** любую <Term id="model">model</Term>, которая была успешно собрана в рамках запуска с использованием оркестрационных возможностей <Constant name="cloud" /> в среде развертывания <Constant name="cloud" />. Модели учитываются в момент их сборки и выполнения. Сюда входят любые задания, запущенные через планировщик <Constant name="cloud" />, CI-сборки (задания, инициированные pull request’ами), запуски, выполненные через API <Constant name="cloud" />, а также любые последующие инструменты <Constant name="cloud" /> с аналогичной функциональностью.  

Также учитываются модели, которые были успешно собраны, даже если сам запуск в итоге не завершился успешно. Например, у вас может быть задание, содержащее 100 моделей, и в одном из запусков 51 модель успешно собирается, после чего задание падает. В такой ситуации будут засчитаны только 51 модель.

Любые модели, собранные в среде разработки <Constant name="cloud" /> (например, через <Constant name="cloud_ide" />), не засчитываются в использование. Тесты, seeds, ephemeral-модели и snapshots также не учитываются.

При первоначальном создании dynamic table модель засчитывается (если создание прошло успешно). Однако при последующих запусках dbt пропускает эти модели, если определение dynamic table не изменилось. Речь идет не об изменениях в SQL-логике, а об изменениях в логике dbt, в частности тех, которые управляются настройкой [`on_configuration_change config`](/reference/resource-configs/on_configuration_change)). Dynamic table продолжает обновляться с заданной периодичностью, поскольку этим обновлением управляет адаптер, а не <Constant name="cloud" />.

| Что учитывается в успешно построенных моделях |                     |
|---------------------------------------------|---------------------|
| View                                        | ✅                  |
| Table                                       | ✅                  |
| Incremental                                 | ✅                  |
| Ephemeral Models                            | ❌                  |
| Tests                                       | ❌                  |
| Seeds                                       | ❌                  |
| Snapshots                                   | ❌                  |

### Что считается запрошенной метрикой?

<Constant name="semantic_layer" />, работающий на базе MetricFlow, учитывает использование в виде отдельных *queried metrics* (запрошенных метрик).

- Каждый успешный запрос, который вы отправляете для рендеринга или выполнения SQL через API <Constant name="semantic_layer" />, считается как минимум одной запрошенной метрикой, даже если в результате не возвращаются данные.  
- Если запрос вычисляет или рендерит SQL сразу для нескольких метрик, каждая вычисленная метрика будет учитываться как отдельная запрошенная метрика.  
- Если запрос на выполнение не был успешно выполнен в платформе данных или если выполнение запроса завершилось ошибкой без успешного завершения, он не учитывается как запрошенная метрика.  
- Запросы метаданных из <Constant name="semantic_layer" /> также не считаются запрошенными метриками.

Примеры запрошенных метрик включают:

- Запрос одной метрики с группировкой по одному измерению → 1 запрошенная метрика 

  ```shell
  dbt sl query --metrics revenue --group-by metric_time
  ```

- Запрос одной метрики с группировкой по двум измерениям → 1 запрошенная метрика 

  ```shell
  dbt sl query --metrics revenue --group-by metric_time,user__country
  ```

- Запрос двух метрик с группировкой по двум измерениям → 2 запрошенные метрики 

  ```shell
  dbt sl query --metrics revenue,gross_sales --group-by metric_time,user__country
  ```

- Запуск компиляции для одной метрики → 1 запрошенная метрика

  ```shell
  dbt sl query --metrics revenue --group-by metric_time --compile
  ```

- Запуск компиляции для двух метрик → запрашиваются 2 метрики

  ```shell
  dbt sl query --metrics revenue,gross_sales --group-by metric_time --compile
  ```

## dbt Copilot: учёт и ограничение использования <Lifecycle status="Starter, Enterprise, Enterprise+" />

Использование <Constant name="copilot" /> измеряется по количеству завершённых AI‑запросов, называемых действиями <Constant name="copilot" />. Ограничения использования применяются для обеспечения справедливого доступа и стабильной работы системы.

Определённое количество вызовов <Constant name="copilot" /> выделяется ежемесячно в зависимости от вашего [тарифного плана](https://www.getdbt.com/pricing). После достижения лимита использование функциональности Copilot будет временно отключено до начала следующего биллингового цикла.

## Планы и выставление счетов

dbt Cloud предлагает несколько [планов](https://www.getdbt.com/pricing) с различными функциями, которые соответствуют вашим потребностям. Мы можем вносить изменения в детали наших планов время от времени. Мы всегда сообщим вам об этом заранее, чтобы вы могли подготовиться. Следующий раздел объясняет, как работает выставление счетов в каждом плане.

<Constant name="cloud" /> предлагает несколько [планов](https://www.getdbt.com/pricing) с различным набором возможностей, которые подойдут под ваши потребности. Время от времени мы можем вносить изменения в детали планов. Мы всегда будем заранее уведомлять вас об этом, чтобы вы могли подготовиться. В следующем разделе объясняется, как работает биллинг в каждом плане.

Планы Developer бесплатны и включают одну лицензию Developer и 3000 моделей каждый месяц. Модели обновляются в начале каждого календарного месяца. Если вы превысите 3000 моделей, любые последующие запуски будут отменены до обновления моделей или до обновления на платный план. Остальная часть платформы dbt Cloud остается доступной, и никакая работа не будет потеряна.

Планы Developer предоставляются бесплатно и включают одну лицензию Developer и 3 000 моделей в месяц. Модели обновляются в начале каждого календарного месяца. Если вы превысите лимит в 3 000 моделей, все последующие запуски будут отменяться до тех пор, пока модели не будут обновлены или пока вы не перейдёте на платный план. При этом остальная часть платформы <Constant name="cloud" /> остаётся доступной, и никакая работа не будет потеряна.

### Выставление счетов по плану Team

Клиенты плана Team платят ежемесячно с помощью кредитной карты за места и использование, и учетные записи включают 15 000 моделей ежемесячно. Места оплачиваются заранее в начале месяца. Если вы добавите места в течение месяца, они будут пропорционально рассчитаны и оплачены в тот же день. Места, удаленные в течение месяца, будут отражены в следующем счете и не подлежат возврату. Вы можете изменить информацию о кредитной карте и количество мест в разделе выставления счетов в любое время. Учетные записи будут получать один ежемесячный счет, который включает авансовый платеж за места и использование, начисленное за предыдущий месяц.

### Выставление счетов для тарифа Starter

Клиенты на тарифе Starter оплачивают места (seats) и использование сервиса ежемесячно с помощью кредитной карты, при этом в аккаунт включено 15 000 моделей в месяц. Стоимость мест списывается авансом в начале месяца. Если вы добавляете места в течение месяца, их стоимость рассчитывается пропорционально и списывается в тот же день. Места, удалённые в течение месяца, будут учтены в следующем счёте и не подлежат возврату. Вы можете в любое время изменить данные кредитной карты и количество мест в разделе выставления счетов (billing). Аккаунты получают один ежемесячный счёт, который включает авансовую оплату за места и оплату за использование сервиса, начисленную задним числом за предыдущий месяц.

`($100 x количество мест для разработчиков) + ((построенные модели - 15,000) x $0.01)`

Все включенные успешные построенные модели, указанные выше, отражают наше самое актуальное ценообразование и упаковку. В зависимости от условий использования, когда вы подписались на план Team, включенные права на модели могут отличаться от указанных выше.

### Выставление счетов по плану Enterprise

Как клиент Enterprise, вы платите ежегодно по счету, ежемесячно за дополнительное использование (если применимо) и можете воспользоваться согласованными ставками использования. Пожалуйста, обратитесь к вашему заказу или контракту для получения конкретных деталей ценообразования или [свяжитесь с командой аккаунта](https://www.getdbt.com/contact-demo) с любыми вопросами.

Все указанные выше количества успешно собранных моделей отражают наши самые актуальные условия ценообразования и пакеты. В зависимости от условий использования, с которыми вы согласились при подключении тарифа Starter, включённые лимиты на модели могут отличаться от тех, что указаны выше.

### Устаревшие планы

Клиенты, которые приобрели план dbt Cloud Team до 11 августа 2023 года, остаются на устаревшем плане ценообразования, пока ваша учетная запись находится в хорошем состоянии. Устаревший план ценообразования основан на местах и включает неограниченные модели, при условии разумного использования.

Информация о биллинге плана Enterprise недоступна в интерфейсе <Constant name="cloud" />. Все изменения обрабатываются через вашего Solutions Architect в dbt Labs или менеджера по работе с аккаунтом.

### Legacy plans

Клиенты, которые приобрели план <Constant name="cloud" /> Starter (ранее Team) до 11 августа 2023 года, остаются на устаревшем тарифном плане при условии, что их аккаунт находится в хорошем состоянии. Устаревший тарифный план основан на количестве мест (seats) и включает неограниченное количество моделей при соблюдении принципа разумного использования.

:::note Legacy <Constant name="semantic_layer" />

Для клиентов, использующих устаревший <Constant name="semantic_layer" /> с пакетом dbt_metrics, этот продукт будет выведен из эксплуатации в декабре 2023 года. Пользователи устаревшей версии могут в любой момент перейти на обновлённую версию — <Constant name="semantic_layer" />, работающую на базе MetricFlow. Обновлённая версия доступна большинству клиентов (см. [prerequisites](/guides/sl-snowflake-qs#prerequisites)) в течение ограниченного времени в рамках бесплатного пробного периода при соблюдении принципа разумного использования.

:::

dbt Labs может установить ограничения на использование, если разумное использование превышено. Дополнительные функции, обновления или обновления могут быть предметом отдельных сборов. Любые изменения в текущем ценообразовании вашего плана будут сообщены заранее в соответствии с нашими Условиями использования.

## Управление использованием

В dbt Cloud нажмите на имя вашей учетной записи в левом меню и выберите **Account settings**. Опция **Billing** будет в левом меню под заголовком **Settings**. Здесь вы можете просмотреть доступные индивидуальные планы и предоставляемые для каждого из них функции.

В <Constant name="cloud" /> нажмите на имя вашей учетной записи в левом боковом меню и выберите **Account settings**. Пункт **Billing** будет доступен в левом боковом меню в разделе **Settings**. Здесь вы можете просмотреть доступные тарифные планы и функции, которые предоставляются в рамках каждого из них.

Каждый план автоматически отправляет уведомления по электронной почте, когда достигнуто 75%, 90% и 100% оценок использования. В плане Team все пользователи в группе владельцев будут получать уведомления. В планах Enterprise все пользователи с разрешениями администратора учетной записи и администратора выставления счетов будут получать уведомления. Пользователи не могут отказаться от этих писем. Если вы хотите, чтобы дополнительные пользователи получали эти уведомления по электронной почте, предоставьте им соответствующие разрешения, упомянутые выше. Обратите внимание, что ваше использование может уже быть выше, чем процент, указанный в уведомлении, из-за вашего шаблона использования и незначительных задержек.

Каждый тариф автоматически отправляет email‑уведомления при достижении **75%**, **90%** и **100%** от оценочного объёма использования.
- **Starter plan** — уведомления получают все пользователи из группы **Owner**. 
- **Enterprise‑tier plans** — уведомления получают все пользователи с наборами прав **Account Admin** и **Billing Admin** ([permission sets](/docs/cloud/manage-access/enterprise-permissions#permission-sets)). 

Пользователи не могут отказаться от получения этих писем. Чтобы дополнительные пользователи также получали такие уведомления, назначьте им соответствующие права, упомянутые выше. Обратите внимание, что фактическое использование может уже превышать процент, указанный в уведомлении, из‑за характера потребления ресурсов и небольшой задержки в обновлении данных.

Есть 2 варианта отключения построения и начисления моделей:

1. Откройте **Job Settings** каждого задания и перейдите в раздел **Triggers**. Отключите **Run on Schedule** и установите функцию **Continuous Integration** **Run on Pull Requests?** в **No**. Проверьте свои рабочие процессы, чтобы убедиться, что вы не запускаете никакие запуски через API dbt Cloud. Этот вариант позволит вам сохранить ваши задания dbt Cloud без построения дополнительных моделей.
2. В качестве альтернативы, вы можете удалить некоторые или все ваши задания dbt Cloud. Это гарантирует, что никакие запуски не будут инициированы, но вы навсегда потеряете свои задания.

1. Откройте **Job Settings** каждой задачи и перейдите в раздел **Triggers**. ## Оптимизация затрат в dbt

<Constant name="cloud" /> предоставляет способы оптимизировать использование сборок моделей и затраты на хранилище данных.

#### Исключение представлений в задании dbt Cloud

Когда вы задумываетесь о способах оптимизации затрат для уже успешно построенных моделей, существуют методы, которые позволяют снизить эти затраты, при этом продолжая следовать лучшим практикам. Чтобы гарантировать, что вы по‑прежнему используете тесты и пересобираете представления (views) при изменении логики, рекомендуется внедрять комбинацию best practices, которая подходит именно под ваши требования. В частности, если вы решаете исключить представления из регулярно запланированных запусков заданий <Constant name="cloud" />, крайне важно настроить merge‑job (со ссылкой на соответствующий раздел), который будет деплоить обновлённую логику представлений при обнаружении изменений.

#### Исключение представлений (views) в dbt‑job

Многие пользователи <Constant name="cloud" /> используют представления, которые не всегда требуется пересобирать при каждом запуске job. Для заданий, которые содержат представления и которые:

- _не_ используют макросы, динамически генерирующие код (например, `case`‑выражения) на основе вышестоящих таблиц, и  
- _не_ имеют тестов,

вы можете выполнить следующие шаги:

1. Перейдите к вашему текущему production deployment job в <Constant name="cloud" />.
2. Измените команду, добавив параметр: `--exclude config.materialized:view`.
3. Сохраните изменения job.

Запуск тестов для представлений в каждом запуске задания может помочь поддерживать качество данных и избавить вас от необходимости повторного запуска неудачных заданий. Чтобы исключить представления из вашего запуска задания при запуске тестов, вы можете выполнить следующие шаги, чтобы создать пользовательский [селектор](https://docs.getdbt.com/reference/node-selection/yaml-selectors) для вашей команды задания.

#### Исключение представлений (views) при запуске тестов

Запуск тестов для представлений при каждом выполнении задания может помочь поддерживать качество данных и избавить от необходимости повторно запускать упавшие задания. Однако, если вы хотите исключить представления из выполнения задания при запуске тестов, вы можете выполнить следующие шаги и создать пользовательский [selector](/reference/node-selection/yaml-selectors) для команды вашего задания.

1. Откройте ваш dbt‑проект в <Constant name="cloud_ide" />.
2. Добавьте файл `selectors.yml` в корневую папку проекта.
3. В этом файле добавьте следующий код:

   ```yaml 
    selectors:
      - name: skip_views_but_test_views
        description: >
          A default selector that will exclude materializing views
          without skipping tests on views.
        default: true
        definition:
          union:
            - union: 
              - method: path
                value: "*"
              - exclude: 
                - method: config.materialized
                  value: view
            - method: resource_type
              value: test

    ```
    
4. Сохраните файл и закоммитьте его в ваш проект.  
5. Измените ваши dbt‑джобы так, чтобы они включали `--selector skip_views_but_test_views`.

#### Построение только измененных представлений

Если вы хотите убедиться, что вы строите представления всякий раз, когда логика изменяется, создайте задание слияния, которое запускается при слиянии кода в основную ветку:

1. Убедитесь, что в вашей среде настроена [CI job](/docs/deploy/ci-jobs).
2. Создайте новую [deploy job](/docs/deploy/deploy-jobs#create-and-schedule-jobs) и назовите её «Merge Job».
3. Установите значение **Environment** равным вашему CI‑окружению. Подробнее см. в разделе [Types of environments](/docs/deploy/deploy-environments#types-of-environments).
4. Установите **Commands** в значение: `dbt run -s state:modified+`.
   
   Выполнять `dbt build` в этом контексте не требуется, так как CI job уже использовалась для выполнения и тестирования кода, который только что был смержен в `main`.
5. В разделе **Execution Settings** выберите стандартную production job, с которой будут сравниваться изменения:
    - **Defer to a previous run state** &mdash; выберите созданную вами «Merge Job», чтобы job сравнивала и определяла, что изменилось с момента последнего merge.
6. В вашем dbt‑проекте выполните шаги из раздела *Run a <Constant name="cloud" /> job on merge* в руководстве [Customizing CI/CD with custom pipelines](/guides/custom-cicd-pipelines), чтобы создать скрипт, который будет вызывать API <Constant name="cloud" /> для запуска вашей job после выполнения merge в git‑репозитории, либо посмотрите это [видео](https://www.loom.com/share/e7035c61dbed47d2b9b36b5effd5ee78?sid=bcf4dd2e-b249-4e5d-b173-8ca204d9becb).

Цель задания слияния:

- Немедленно развернуть любые изменения из PR в производство.
- Обеспечить, чтобы ваши производственные представления оставались актуальными в соответствии с тем, как они определены в вашем коде, при этом оставаясь экономически эффективными при запуске заданий в производстве.

Действие слияния оптимизирует ваши расходы на облачную платформу данных и сокращает время выполнения заданий, но вам нужно будет решить, подходит ли это изменение для вашего проекта dbt.

### Переработка неэффективных моделей

#### Вкладка Job Insights

Чтобы сократить ваши расходы на хранилище, вы можете определить, какие модели в среднем занимают больше всего времени на построение на странице **Job** во вкладке **Insights**. Этот график рассматривает среднее время выполнения для каждой модели на основе последних 20 запусков. Любые модели, которые занимают больше времени, чем ожидалось, на построение, могут быть основными кандидатами на оптимизацию, что в конечном итоге сократит расходы на облачное хранилище.

#### Вкладка Model Timing

Чтобы лучше понять, сколько времени занимает выполнение каждой модели в контексте конкретного запуска, вы можете посмотреть вкладку **Model Timing**. Выберите интересующий вас запуск на странице **Run History**, чтобы найти вкладку. На этой странице **Run** нажмите **Model Timing**.

## Часто задаваемые вопросы

* **Что произойдёт, если мне потребуется больше мест (seats) на плане Starter?**  
_Если вам нужно больше мест для разработчиков, выберите опцию [Contact Sales](https://www.getdbt.com/contact) в настройках биллинга, чтобы связаться с нашей командой продаж и обсудить план Enterprise или Enterprise+._

* **Что если я значительно превышу количество включённых бесплатных моделей на плане Starter или Developer?**  
_Рассмотрите возможность перехода на план уровня Starter или Enterprise. Планы Starter и Enterprise включают большее количество моделей и позволяют превышать месячный лимит использования. Аккаунты Enterprise сопровождаются выделенной командой по управлению аккаунтом и предлагают годовые контракты, пользовательские конфигурации и согласованные тарифы использования._

* **Я хочу обновить свой план. Перенесётся ли вся моя работа?**  
_Да. Ваш аккаунт <Constant name="cloud" /> будет обновлён без влияния на существующие проекты и настройки аккаунта._

* **Как определить, какой план мне подходит?**  
_Лучший вариант — проконсультироваться с нашей командой продаж. Они помогут подобрать оптимальное решение под ваши задачи. Также мы предлагаем бесплатный двухнедельный пробный период на плане Starter._

* **Каковы условия пробного периода для <Constant name="semantic_layer" />?**  
_Клиенты планов Starter и Enterprise могут зарегистрироваться на бесплатный пробный период <Constant name="semantic_layer" />, работающего на MetricFlow, с использованием до 1 000 Queried Metrics в месяц. Пробный период будет доступен как минимум до января 2024 года. dbt Labs может продлить пробный период по своему усмотрению. В течение пробного периода мы можем связаться с вами, чтобы обсудить варианты ценообразования или запросить обратную связь. По окончании пробного периода бесплатный доступ может быть отключён, и для продолжения использования может потребоваться покупка. dbt Labs оставляет за собой право изменять лимиты в рамках бесплатного пробного периода или вводить платное использование при необходимости либо в любое время по своему усмотрению._

* **Какое ограничение разумного использования действует для <Constant name="semantic_layer" />, работающего на MetricFlow, в течение пробного периода?**  
_В течение пробного периода каждый аккаунт будет ограничен 1 000 Queried Metrics в месяц; это ограничение может быть изменено по собственному усмотрению dbt Labs._
